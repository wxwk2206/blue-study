# 1.对比

| 反射型            | 存储型           | DOM 型             |
| -------------- | ------------- | ----------------- |
| 恶意代码在 URL 中    | 恶意代码在 数据库 中   | 恶意代码在 浏览器前端       |
| 服务器返回 HTML 时嵌入 | 服务器读取后嵌入 HTML | JS 动态修改 DOM 时嵌入   |
| 需要用户点击链接       | 任何访问者自动触发     | URL / Fragment 触发 |
| 服务器端可见         | 服务器端可见        | 服务器完全不可见          |
# 2.概括
DOM 型 XSS（DOM-based Cross-Site Scripting）
漏洞完全发生在浏览器端——JavaScript 从不可信来源（如 URL 参数）读取数据，然后直接拼入页面 DOM，恶意代码在客户端执行，整个过程服务器完全不知情。


**两个核心概念：**

| 概念           | 英文     | 含义                       | 类比       |
| ------------ | ------ | ------------------------ | -------- |
| Source（用户输入） | Source | JavaScript 读取不可信数据的地方    | 你从哪里拿到的信 |
| Sink（危险执行）   | Sink   | JavaScript 把数据写入 DOM 的地方 | 你把信贴在了哪里 |
```
# ⭐ 典型的 DOM 型 XSS 漏洞代码（前端 JavaScript）

# 页面 HTML 中有这样一段 JS：
<script>
  // 从 URL 的 hash（#号后面）读取内容
  var userInput = location.hash.substring(1);

  // ⚠️ 直接用 innerHTML 写入页面——这就是 Sink！
  document.getElementById('output').innerHTML = '欢迎，' + userInput;
</script>

# 正常访问
https://example.com/page#Alice
→ 页面显示：欢迎，Alice  ✅ 正常

# ⭐ DOM 型 XSS 攻击
https://example.com/page#<img src=x onerror=alert(1)>
→ JS 读取 hash 内容：
→ innerHTML 写入 DOM
→ 浏览器执行 onerror → 弹窗！
→ ⭐ 服务器收到的请求完全正常，返回的 HTML 也完全正常
```
# 3.Source 和 Sink 速查表
## 3.1 Source 列表——不可信数据来源
| Source              | 说明             | 示例                              |
| ------------------- | -------------- | ------------------------------- |
| `location.href`     | 完整 URL         | `https://evil.com/page#payload` |
| `location.search`   | URL 的 `?` 之后部分 | `?name=<script> ... </script>`  |
| `location.hash`     | URL 的 `#` 之后部分 | `#<img src=x onerror=alert(1)>` |
| `location.pathname` | URL 路径部分       | `/page/<script> ... </script>`  |
| `document.referrer` | 来源页面 URL       | 可伪造的 Referer 头                  |
| `document.cookie`   | Cookie 值       | 如果 Cookie 包含用户可控内容              |
| `window.name`       | 窗口名称           | 跨域可控                            |
| `postMessage` 数据    | 跨窗口消息          | 来自不可信 origin 的消息                |
## 3.2Sink 列表——危险执行点
| Sink                     | 危险等级 | 说明                    |
| ------------------------ | ---- | --------------------- |
| `innerHTML`              | 极高   | 直接解析 HTML，脚本自动执行      |
| `outerHTML`              | 极高   | 同上，替换整个元素             |
| `document.write()`       | 极高   | 写入 HTML 文档流           |
| `document.writeln()`     | 极高   | 同上                    |
| `eval()`                 | 极高   | 执行任意 JS 代码            |
| `setTimeout(string)`     | 高    | 字符串参数会被 eval          |
| `setInterval(string)`    | 高    | 同上                    |
| `Function(string)`       | 高    | 动态创建函数 = 执行代码         |
| `element.src`            | 中    | 如设为 `javascript: ...` |
| `element.href`           | 中    | 如设为 `javascript: ...` |
| `element.setAttribute()` |  视情况 | 设置事件属性时危险             |
**安全替代：**
textContent 和 innerText 是安全的——它们只设置纯文本，不会解析 HTML。innerHTML 永远不应用于展示用户可控数据。
# 4.实战流程
**1.审查前端 JS 代码**
按 F12 打开开发者工具 → Sources 面板，搜索所有 Sink 关键字：innerHTML、document.write、eval 等。
**2.追踪数据来源**
找到 Sink 后，往回追踪数据来自哪个 Source：location.search、location.hash、document.referrer 等。
**3.检查中间是否有过滤/净化**
看 Source → Sink 之间有没有 textContent、encodeURIComponent、输入验证、白名单过滤等安全措施。
**4.构造 Payload 验证**
在 URL 的对应位置插入恶意值，验证是否能触发 XSS。

# 5.常见 DOM 型 XSS Payload
Payload 本身和反射型/存储型相同，但注入位置不同——放在 URL 参数中。

| 注入位置 | 完整 URL 示例 | 触发机制 |
| ---- | ---- | ---- |
| URL search 参数 | `https://target.com/?q=<img src=x onerror=alert(1)>` | `location.search` → `innerHTML` |
| URL hash | `https://target.com/page#<svg/onload=alert(1)>` | `location.hash` → `innerHTML` |
| URL 路径 | `https://target.com/<script>alert(1)</script>` | `location.pathname` → `document.write` |
| eval 注入 | `https://target.com/?callback=alert(1)//` | `location.search` → `eval()` |
| template literal | `https://target.com/?name=${alert(1)}` | `location.search` → ES6 模板字符串 → DOM |
 **URL hash 的特殊性**
   \# 后面的内容不会发送到服务器。这意味着：
1. 服务器端安全措施（WAF、输入过滤）全部无效
2. 服务器日志里看不到任何痕迹
3. 但也意味着必须让用户点击完整 URL（含 hash）才能触发
