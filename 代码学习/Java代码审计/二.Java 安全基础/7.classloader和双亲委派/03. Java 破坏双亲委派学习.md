# 0x01 概述
```plain
双亲委派模型并不是一个具有强制性约束的模型
而是Java设计者推荐给开发者们的类加载器实现方式
这个委派和加载顺序完全是可以被破坏的

如果想自定义类加载器,就需要继承ClassLoader,并重写findClass方法
如果想不遵循双亲委派的类加载顺序,就需要继承ClassLoader,并重写findClass与loadClass方法
```

# 0x02 为什么要破坏双亲委派?
```plain
现在我们知道class文件的加载是按照双亲委托机制完成的
这个机制解决了各个类加载器的基础类的统一问题
因为上层类加载器加载的类对下层加载的类是可见的
所以这些基础类可以被Java程序所调用

但是如果这些基础类需要调用用户所写的类呢?
这种情况不是不可能的,比如JDBC,它位于rt.jar包下,是由引导类加载器加载
而它却需要ClassPath下的接口提供者(Service Provider Interface)的代码
但是引导类加载器不可能去加载这些代码,那怎么办呢?
这时候就需要对双亲委托机制进行“破坏”了
```

## 0x03.2 自定义类加载器
### 0x03.2.1 自定义加载器的注意项学习
```java
// 先创建一个TestCustomClassLoader类,这是个自定义的类加载器,并重写findClass和loadClass方法
// 路径: ./ClassLoaderLearning/src/main/java/study/custom/classLoader/test/TestCustomClassLoader.java
package study.custom.classLoader.test;

import java.io.File;
import java.nio.file.Files;
import java.nio.file.Path;
import java.nio.file.Paths;
import java.io.IOException;

public class TestCustomClassLoader extends ClassLoader {
    public TestCustomClassLoader() {
        super(TestCustomClassLoader.class.getClassLoader());
    }

    public TestCustomClassLoader(ClassLoader parent) {
        super(parent);
    }

    @Override
    protected Class findClass(String name) throws ClassNotFoundException {
        // 定义class文件的相对路径
        String dir = System.getProperty("user.dir");
        String classPath = dir + File.separator + "target" + File.separator + "classes" + File.separator;

        // 获取class文件二进制字节数组
        byte[] data = null;
        try {
            String namePath = name.replaceAll("\\.", File.separator);
            String classFile = classPath + namePath + ".class";
            Path path = Paths.get(classFile);
            data = Files.readAllBytes(path);
        } catch (IOException e) {
            e.printStackTrace();
        }

        // 字节码加载到JVM的方法区,并在JVM的堆区建立一个java.lang.Class对象的实例
        return this.defineClass(name, data, 0, data.length);
    }

    @Override
    public Class loadClass(String name) throws ClassNotFoundException {
        Class clazz = null;

        // 直接自己加载
        clazz = this.findClass(name);
        if (clazz != null) {
            return clazz;
        }

        // 自己加载不了,再调用父类loadClass,保持双亲委托模式
        return super.loadClass(name);
    }
}
```

```plain
运行“./study/custom/classLoader/test/Test.java”测试代码,发现报错了
报错信息为:“/java/lang/Object.class”

问题来了,我加载“study.custom.classLoader.Demo.java”和“Object.class”有什么关系呢?
答案很简单,因为Object类是所有类的基类
当一个类没有直接继承某个类时,默认继承Object类,也就是说任何类都直接或间接继承Object类

因此加载“study.custom.classLoader.Demo.java”时,也会加载Object这个父类
Object这个父类,存在于“/jre/lib/rt.jar”(JDK内部的包)
TestCustomClassLoader类的class加载路径被固定在“/ClassLoaderLearning/target/classes/”
也就是说Object类与TestCustomClassLoader类的class加载路径,并不在同个目录
```

## 0x03.3 <font style="color:rgb(79, 79, 79);">线程上下文类加载器</font>
### 0x03.3.1 概述
```plain
线程上下文类加载器其实是一种类加载器传递机制
可以通过“java.lang.Thread#setContextClassLoader”方法,给一个线程设置上下文类加载器
在通过“java.lang.Thread#getContextClassLoader”方法,获取这个上下文类加载器
如果在应用程序的全局范围内都没有设置过就默认是ApplicationClassLoader(应用程序类加载器)

注意: 线程上下文类加载器的出现就是为了方便破坏双亲委派
```

```java
// 路径: ./ClassLoaderLearning/src/main/java/study/thread/classLoader/test5/Test5.java
package study.thread.classLoader.test5;

public class Test5 {
    public static void main(String[] args) throws ClassNotFoundException {
        // 获取使用当前类的ClassLoader
        ClassLoader classLoader = Test5.class.getClassLoader();
        System.out.println("当前类的ClassLoader是:" + classLoader);

        // 获得当前线程上下文类加载器
        ClassLoader classLoader2 = Thread.currentThread().getContextClassLoader();
        System.out.println("当前线程的ClassLoader是:" + classLoader2);

        try {
            // 设置当前线程上下文类加载器为ExtentionClassLoader(扩展类加载器)
            ClassLoader extentionClassloader = ClassLoader.getSystemClassLoader().getParent();
            Thread.currentThread().setContextClassLoader(new Test5CustomClassLoader(extentionClassloader));

            System.out.println("现在线程的ClassLoader是:" + Thread.currentThread().getContextClassLoader());

            // 通过当前线程上下文类加载器,加载Demo3类
            Class clazz = Thread.currentThread().getContextClassLoader().loadClass("study.thread.classLoader.Demo3");
            System.out.println("Demo3的类加载器是: " + clazz.getClassLoader());
            System.out.println("Demo3的父类加载器是: " + clazz.getClassLoader().getParent());
        } finally {
            // 还原当前线程上下文类加载器
            Thread.currentThread().setContextClassLoader(classLoader2);
        }

        // 重新获得当前线程上下文类加载器,查看是否还原
        System.out.println("还原以后线程的ClassLoader是:" + Thread.currentThread().getContextClassLoader());
    }
}
```

# 0x04 小结
```plain
1. 官方库“java.”开头的类必须由引导类加载器加载,无法破坏
2. 扩展类加载器和应用程序类加载器的双亲委派都是可以破坏的
3. 破坏双亲委派的方法有自定义类加载器与线程上下文类加载器
```
