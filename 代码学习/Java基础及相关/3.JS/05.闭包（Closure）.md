# 1.前言
一个诡异现象：
```js
function createCounter() {
  let count = 0;                    // 在函数里定义了一个局部变量

  return function() {               // 返回一个内部函数
    count = count + 1;              // 内部函数访问了外部变量 count
    return count;
  };
}

const counter = createCounter();    // createCounter 执行完了
console.log(counter());  // → ?
console.log(counter());  // → ?
console.log(counter());  // → ?
```
按照传统语言（Java/C）的逻辑，createCounter 执行完后，局部变量 count 应该被销毁。但实际输出是：

count 不仅没被销毁，还在持续累加。这就是**闭包**——内部函数「记住」了它定义时所在的作用域，即使外层函数已经执行完毕。
# 2.闭包的定义
概括：闭包 = 函数 + 该函数定义时所在的词法环境。函数「记住」并可以访问它定义时的作用域，哪怕在作用域外被调用。
## 形成闭包的三个条件
- 存在**嵌套函数**（函数里有函数）
- 内部函数**引用了外部函数的变量**
- 内部函数被**返回到外部**（或以其他方式暴露）
```js
function outer() {
  const outerVar = '我是外部变量';

  function inner() {           // ① 嵌套函数
    console.log(outerVar);     // ② 引用了外部变量
  }

  return inner;                // ③ 返回到外部
}

const fn = outer();   // outer 执行完，但 outerVar 被闭包保留
fn();                 // → '我是外部变量'
```
⚠️ 注意：闭包不是「创建函数」这个动作，而是**函数本身 + 它携带的作用域引用**。可以说 inner 就是一个闭包。每次调用 outer() 都会创建一个**新的闭包**，各自独立。

# 3.JS 有闭包的原因
根因是 JS 的词法作用域（Lexical Scope）：变量的作用域在**代码写出来时就决定了**，由函数定义的位置决定，而不是调用时决定。
```js
// 词法作用域：函数能看到什么变量，看它在哪定义，不是在哪调用
const globalMsg = '全局';

function outer() {
  const outerMsg = '外层';

  function inner() {
    // inner 定义在 outer 内部
    // 所以它能访问 outerMsg 和 globalMsg
    console.log(globalMsg, outerMsg);
  }

  return inner;
}

// 把 inner 带到完全不同的环境再调用
const fn = outer();
fn();  // → '全局' '外层'  ← 依然记得自己来自哪
```
一句话：JS 函数随身携带自己定义时的环境。这个「环境引用」就是闭包的本质。
# 4.应用场景
## 4.1 私有变量（模块模式）
JS 没有真正的 private 关键字（class 的 # 私有字段是新特性），闭包是实现私有变量的传统方式：
```js
// ⭐ 模块模式：用闭包隐藏内部状态，只暴露公共接口
function createBankAccount(initialBalance) {
  let balance = initialBalance;   // 私有变量，外部访问不到

  return {
    deposit(amount) {              // 公共方法
      balance += amount;
      return balance;
    },
    withdraw(amount) {             // 公共方法
      if (amount > balance) return '余额不足';
      balance -= amount;
      return balance;
    },
    getBalance() {                 // 公共方法
      return balance;
    }
  };
}

const acc = createBankAccount(100);
console.log(acc.getBalance());    // → 100
console.log(acc.deposit(50));     // → 150
console.log(acc.balance);         // → undefined ⚠️ 外部访问不到私有变量
console.log(acc.withdraw(1000));  // → '余额不足'
```

> [!important]  安全视角
> 这就是「封装」。敏感数据（密码、token、余额）藏进闭包，外部只能通过你提供的方法间接操作。做代码审计时，看到这种模式意味着开发者有意控制了访问边界。

## 4.2 函数工厂（柯里化 Currying）

```js
// ⭐ 用闭包「记住」一部分参数，生成专用函数
function createGreeter(greeting) {
  return function(name) {
    return `${greeting}, ${name}!`;
  };
}

const sayHello = createGreeter('Hello');
const sayHi    = createGreeter('Hi');

console.log(sayHello('bluek'));  // → 'Hello, bluek!'
console.log(sayHi('bluek'));     // → 'Hi, bluek!'
// sayHello 记住了 greeting='Hello'，sayHi 记住了 greeting='Hi'
```

## 4.3 回调与事件处理（最常见）
```js
// ⭐ 异步回调天然形成闭包
function fetchData(url) {
  const requestId = Math.random();   // 每次调用生成不同 ID

  fetch(url).then(function(response) {
    // 这个回调函数闭包住了 requestId
    // 即使 fetch 完成时 fetchData 早就执行完了
    console.log('请求 ID:', requestId, '完成');
  });
}
```

# 5.闭包陷阱：循环中的闭包
这是 JS 最经典的面试题，也是真实的 bug 源头。先看问题代码：
```js
// ⚠️ 用 var 的灾难现场
for (var i = 1; i <= 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, 100);
}
// 你以为输出：1 2 3
// 实际输出：  4 4 4  ← 全是 4！
```
原因分析：
1. var 没有块级作用域，整个循环只有一个 i
2. 三个 setTimeout 的回调函数**闭包的是同一个 i**
3. 循环结束时 i = 4（i++ 到 4 才退出）
4. 100ms 后回调执行，读到的 i 都是 4
## 5.1 修复方案一：用 let（推荐）
```js
// ⭐ let 有块级作用域，每次循环都是一个全新的 i
for (let i = 1; i <= 3; i++) {
  setTimeout(function() {
    console.log(i);   // → 1, 2, 3 ✅
  }, 100);
}
```
## 5.2 修复方案二：用 IIFE 创建独立作用域
```js
// ⭐ 老代码常用：IIFE 把当时的 i 复制一份
for (var i = 1; i <= 3; i++) {
  (function(j) {              // IIFE 接收 i 作为参数 j
    setTimeout(function() {
      console.log(j);          // → 1, 2, 3 ✅
    }, 100);
  })(i);
}
```

> [!important]  代码审计视角
> 老代码里 for + var + 异步回调 是经典 bug 模式。看到这种结构要警惕：循环里的闭包可能都引用同一个变量，导致逻辑错误。这种 bug 不会抛异常，但行为完全错乱——在安全场景里可能表现为权限检查错位、数据混淆。

# 6.闭包的代价：内存
闭包会保留外部变量，这意味着这些变量不会被垃圾回收（Garbage Collection, GC）。
```js
// ⚠️ 闭包持有大对象，导致内存无法释放
function createLeak() {
  const hugeData = new Array(1000000).fill('数据');  // 占用大量内存

  return function() {
    console.log('我还活着');
    // 这个函数没用 hugeData，但闭包依然保留了它
  };
}

const leak = createLeak();
// hugeData 本该被回收，但因为闭包持有作用域，它一直占着内存
```
⚠️ 注意：现代 JS 引擎会做优化——如果闭包函数没引用某个变量，引擎可能不会保留它。但不要依赖这个优化，写代码时注意闭包里不要引用不必要的大对象。

> [!important]  安全相关
> - 内存泄漏（Memory Leak）可能导致 DoS——服务器慢慢吃光内存
> - 逆向时，闭包里的变量无法通过常规手段访问，但可以通过 DevTools 的 Scope 面板查看
> - 有些恶意代码故意用闭包藏数据（密钥、配置），增加分析难度
# 7.闭包在 JS 逆向中的意义
混淆代码大量使用闭包来隐藏逻辑。看个简化示例：
```js
// ⭐ 典型的混淆模式：IIFE + 闭包 + 返回核心函数
var _0x4a2b = (function() {
  var _key = 'a3f5b8c9d2e1f0';           // 密钥藏在闭包里
  var _config = { salt: 'xyz', rounds: 10 };

  function _encrypt(data) {               // 内部函数
    // 用 _key 和 _config 加密 data
    return data + _key + _config.salt;
  }

  // 只暴露需要的方法
  return {
    encrypt: _encrypt
    // _key、_config、解密函数都不暴露
  };
})();

// 外部只能用：
console.log(_0x4a4a2b.encrypt('hello'));  // → 'helloa3f5b8c9d2e1f0xyz'

// 但拿不到 _key：
console.log(_0x4a2b._key);  // → undefined
console.log(_key);           // ❌ ReferenceError
```

> [!important]  逆向技巧
> 遇到闭包藏数据时，用 Chrome DevTools → Sources → 在闭包函数内打断点 → 触发执行 → 看 Scope 面板里的 Closure 分组，所有被闭包的变量都在那。

# 8.核心要点

| 概念      | 一句话                          |
| ------- | ---------------------------- |
| 闭包定义    | 函数 + 它定义时的词法环境引用             |
| 形成条件    | 嵌套函数 + 引用外层变量 + 暴露到外部        |
| 核心特性    | 函数「记住」定义时的作用域，哪怕在外部调用        |
| 模块模式    | 用闭包实现私有变量，只暴露公共接口            |
| 循环陷阱    | var 共享变量导致闭包都引用同一个值，用 let 修复 |
| 内存代价    | 闭包持有的变量不会被 GC，可能造成内存泄漏       |
| 🔒 逆向意义 | 闭包藏数据，断点 + Scope 面板可查看       |
