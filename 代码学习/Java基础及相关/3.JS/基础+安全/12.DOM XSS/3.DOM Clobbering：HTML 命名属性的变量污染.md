# 1.DOM Clobbering定义
DOM Clobbering 是浏览器的一种历史遗留行为——如果一个 HTML 元素有 id 或 name 属性，浏览器会以该属性的值作为 key，把这个元素本身作为 value挂到 window 对象上。比如 <img id="x"> → window.x 就是那个 img 元素——不管你的 JS 代码里有没有定义变量 x。
## 1.1 一行代码看懂
下面这行 HTML 会静默修改你的 JS 行为：
```js
<img id="alert" src="x">

<script>
  console.log(alert);         // ⚠️ 不是函数了！是这个 img 元素
  console.log(window.alert);  // ⚠️ 同样被覆盖了
  alert('hello');             // ❌ TypeError: alert is not a function
</script>
```
**任何带 id 或 name 的 HTML 元素都会污染 window 上的同名属性，优先级甚至高于 JS 变量。**
## 1.2 会被覆盖的属性
| 元素类型         | HTML                                  | 覆盖的对象              | 说明                                         |
| ------------ | ------------------------------------- | ------------------ | ------------------------------------------ |
| 任意带 id       | `<div id="x">`                        | `window.x`         | 直接映射为 DOM 元素                               |
| 特定元素带 name   | `<img name="x">`                      | `window.x`         | name 属性也有效（img/form/iframe/embed/object）   |
| `<a>` 带 name | `<a id="x" name="y">`                 | `window.x` 和 `x.y` | ⭐ 第二个 name 成为第一个元素的属性！形成嵌套对象               |
| `<form>` 嵌套  | `<form id="cfg"><input name="admin">` | `window.cfg.admin` | ⭐ form 内的 input 成为 form 的属性，形成 `cfg.admin` |
> [!important]  关键区别
> - `id` 对所有元素生效，直接映射到 `window.xxx`
> - `name` 只在特定元素上生效（img, form, iframe, embed, object, a），但还能形成**嵌套层级**
> - `<a>` 有两个特殊之处：① 它的 `toString()` 返回 `href` 的值（可控字符串！）② 嵌套 name 形成层级

# 2.DOM Clobbering 和 DOM XSS 的关系
它们不是同一种漏洞，但经常**联动**。DOM Clobbering 本身不执行代码——它只能覆盖变量。但覆盖的变量如果被用于安全检查（如 if (config.isAdmin)）或 DOM 操作，就能**为 DOM XSS 铺路。**

| 维度   | DOM Clobbering                                                                                                    | DOM XSS                       |
| ---- | ----------------------------------------------------------------------------------------------------------------- | ----------------------------- |
| 本质   | 变量覆盖（污染window命名空间）                                                                                                | 代码注入（用户数据变成可执行代码）             |
| 触发条件 | HTML元素有id/name，且JS引用了同名变量                                                                                         | 不可信数据进入Sink（innerHTML/eval/…） |
| 直接危害 | 逻辑错误、安全检查绕过、函数被覆盖                                                                                                 | 任意JS代码执行                      |
| 典型联动 | Clobbering 使 `config.isAdmin` 变成 truthy 的 DOM 元素 → `if (config.isAdmin)` 判定通过 → 走管理员分支 → `innerHTML` 不加过滤写入 → XSS | —                             |
联动案例：从 Clobbering 到 XSS
```js
// ⭐ 场景：一个评论页面的 JS
var config = {
  isAdmin: false,      // 普通用户
  sanitize: true       // 默认过滤评论
};

function renderComment(html) {
  if (!config.sanitize) {
    // ⚠️ 管理员可以发 HTML，不过滤
    document.getElementById('comments').innerHTML = html;
  } else {
    document.getElementById('comments').textContent = html;
  }
}

// ⭐ 攻击者注入的 HTML（通过富文本评论）
// <a id="config"><a id="config" name="sanitize" href="x">
// 结果：window.config → a 元素
//        window.config.sanitize → 第二个 a 元素
//        if (!config.sanitize) → a 元素 truthy → ! 转 false → 条件不满足
//        走不到 XSS 分支……
```
**问题案例**：上面的例子用 !config.sanitize 不会进 XSS 分支。但换个写法——如果 JS 检查的是 if (config.isAdmin)，clobbering 设 isAdmin 为 truthy 值后就能绕过。真实场景中，每个 JS 代码的条件分支逻辑不同，攻击者需要具体分析。
# 3.三种典型攻击模式
## 3.1 覆盖简单变量 / 函数
原理：最常见的情况——HTML 元素的 id 和 JS 变量重名，导致变量被替换为 DOM 元素。
```js
// ⭐ 假设页面上有一段 HTML 是用户可控的（如评论、昵称）
// 用户注入：<img id="submitForm">
// 页面原有：
function submitForm() { /* 正常提交逻辑 */ }

// 现在 window.submitForm 不再是函数，是 img 元素！
// <button onclick="submitForm()"> → TypeError: submitForm is not a function
```

| 目标类型 | 示例 | 效果 |
| ---- | ---- | ---- |
| 覆盖函数 | `<img id="validate">` → `validate()` 报错 | 可绕过前端校验 |
| 覆盖变量 | `<div id="DEBUG">` → `DEBUG` 变为 truthy | 可能进入调试模式分支 |
| 覆盖配置 | `<form id="config"><input name="isAdmin">` | `config.isAdmin` = input 元素 → truthy → 绕过权限 |
## 3.2 嵌套对象覆盖（双重 clobbering）
原理：这是 DOM Clobbering 最具威力的一招——利用 HTML 元素的层级关系，覆盖 **对象的属性。**
条件：如果一段 JS 代码做了 obj.prop 这种链式访问，且 obj 和 prop 都可以被命名属性控制，就能实现嵌套覆盖。

| HTML注入 | JS受影响代码 | 结果 |
| ---- | ---- | ---- |
| `<form id="cfg"><input name="isAdmin" value="1"></form>` | `if (cfg.isAdmin) { ... }` | `cfg.isAdmin` = input 元素 → truthy → 绕过！ |
| `<a id="x"><a id="x" name="y" href="http://evil.com"></a></a>` | `if (x.y === 'good') { ... }` | ⭐ `x.y` 是第二个 a 元素，`x.y.toString()` 返回其 href → 可以是任意字符串！ |
| `<form id="opts"><input name="url" value="javascript:alert(1)"></form>` | `location = opts.url;` | `opts.url` = input 元素，但 `opts.url.value` 可控 |
> [!important]  \<a> 标签的特殊性（核心技巧）
> `<a>` 元素在转换为字符串时会返回它的 `href` 属性值。这意味着如果你能通过 `toString()` 或字符串拼接触发转换，就能把 `href` 的内容注入到 JS 逻辑中。这对于绕过 `===` 严格相等检查或字符串比较非常有用。

## 3.3 覆盖 document 上的属性（少见场景）
原理：除了 window 的命名属性映射，HTMLCollection 也会暴露元素的 name / id。比如 document.images 返回所有 img，document.forms 返回所有 form——如果某个表单元素的 id/name 刚好和 JS 引用的属性名冲突，可能产生意外行为。
```js
// ⭐ 如果 JS 中写的是：
var formLen = window.forms ? window.forms.length : 0;
// 攻击者注入：<form id="forms">
// window.forms 变成该 form 元素（不再是 HTMLCollection）
// window.forms.length → undefined（普通 form 无此属性）→ 逻辑错误
```
注意：这里说的是 window.forms（被 clobber），不是 document.forms（DOM API，不能被 id 覆盖）。攻击的核心是利用 JS 代码里没写全限定名（document vs window）的疏忽。
# 4.防御
| 做法                  | 推荐程度 | 为什么                                              |
| ------------------- | ---- | ------------------------------------------------ |
| 避免全局变量              | 最佳   | 用IIFE包裹代码，用`let/const`而非`var`，严格模式下变量不会自动挂window |
| 用`hasOwnProperty`检查 | ✅    | 访问`window.xxx`前先检查它是否真的来自JS而非DOM                 |
| HTML净化              | ✅    | DOMPurify等库会移除危险id/name值（可配置）                    |
| CSP（内容安全策略）         | 辅助   | CSP阻止XSS执行，但不能阻止clobbering本身。仅作为纵深防御             |
| TypeScript          | 现代   | TS编译后的变量不会自动污染window，且在严格模式下效果更好                 |
