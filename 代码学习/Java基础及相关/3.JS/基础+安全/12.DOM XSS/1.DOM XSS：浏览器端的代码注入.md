DOM XSS 是一种**纯客户端**的 XSS——漏洞不在服务端代码里，而在前端 JavaScript 把不可信数据写入 DOM 的过程中。攻击者构造的恶意数据经由浏览器的某个「入口」进入 JS，再被传到某个「危险出口」，中间没有过滤，就变成了可执行的代码。

	正常数据流：Source(用户输入) → 经过过滤器/中转 → Sink(危险执行)

| 类型 | 漏洞位置 | 数据流向 | 典型例子 |
| ---- | -------- | -------- | -------- |
| 反射型 XSS | 服务端 | 请求参数 → 服务端拼 HTML → 响应 | `?q=<script>` 被原样写进搜索结果页 |
| 存储型 XSS | 服务端 | 数据库 → 服务端读取 → 响应 | 留言板存了恶意脚本，别人访问就触发 |
| ⭐ DOM XSS | 客户端 JS | 浏览器 API → JS 处理 → 写入 DOM | `location.hash` → `innerHTML` |
> [!important]  关键认知
> DOM XSS 的服务端响应不含恶意内容——服务端只返回一段正常的 JS。是这段 JS 自己从 location 等浏览器 API 读到攻击者注入的数据，又把它喂给了 innerHTML。所以看服务端代码查不到漏洞，必须审计前端 JS 的数据流。这也是为什么上节笔记学的「找 Sink」思路在这里直接用得上。

# 2.Source → Sink 数据流
DOM XSS 的全部本质就一句话：**不可信数据从 Source 流到 Sink，中间没有净化**。记住常见的 Source 和 Sink，就能快速判断一段 JS 有没有风险。
## 2.1 常见 Source（污染源）
| Source                                | 含义             | 攻击者能否控制?                   |
| ------------------------------------- | -------------- | -------------------------- |
| `location.hash`                       | URL 中 # 后面的部分  | ✅ 可控，且不发给服务端（隐蔽性强）         |
| `location.search`                     | URL 中 ? 后面的查询串 | ✅ 可控                       |
| `location.href` / `location.pathname` | 完整 URL / 路径部分  | ✅ 可控                       |
| `document.referrer`                   | 来源页 URL        | ✅ 可控（攻击者可构造跳转链）            |
| `event.data`（postMessage）             | 跨窗口消息内容        | ✅ 可控（任何页面都能 postMessage）   |
| `localStorage` / `sessionStorage`     | 本地存储           | ⚠️ 间接可控（若存储值本身来自上述 Source） |
## 2.2 常见 Sink（危险汇）
| Sink | 为什么危险 | 危险等级 |
| ---- | ---- | ---- |
| `element.innerHTML` / `outerHTML` | 会把字符串当HTML解析，`<script>`、`<img onerror>` 都能执行 | 🔴 高 |
| `document.write()` | 直接写入文档流，可注入任意HTML/脚本 | 🔴 高 |
| `eval()` / `new Function()` | 把字符串当代码执行 | 🔴 高 |
| `setTimeout(string)` / `setInterval(string)` | 字符串参数会触发 eval 语义 | 🟠 中高 |
| `element.setAttribute('onclick', x)` | 写入事件属性，x 会成为可执行代码 | 🟠 中 |
| `location = x` / `location.assign(x)` | 可跳转到 `javascript:` 伪协议执行代码 | 🟠 中 |
| jQuery `$(html)` / `.html(x)` | 底层就是 innerHTML | 🔴 高 |
 审计口诀：看到 location / referrer / postMessage 等入口数据，顺着变量追踪，看它最终有没有流进上面任何一个 Sink。中间没有 escape / sanitize 就是漏洞。上节笔记的「找 Sink」+ 本节笔记的「找 Source」= DOM XSS 审计的完整方法。
# 3.实战复现：location.hash → innerHTML
## 3.1 有漏洞的代码
```js
// ⭐ 典型 DOM XSS 漏洞代码
function renderWelcome() {
  // Source：从 URL 锚点读取用户名
  const name = location.hash.substring(1) || '访客';
  // Sink：直接写入 innerHTML，没有过滤！
  document.getElementById('greeting').innerHTML = '<h2>欢迎，' + name + '！</h2>';
}
renderWelcome();
```
正常访问 page.html#小明 显示「欢迎，小明！」。但攻击者发你 page.html#\<img src=x onerror=alert(document.cookie)>，innerHTML 会把这段当 HTML 解析，<img> 加载失败触发 onerror，弹出你的 Cookie。
## 3.2 漏洞分析（原理 → 条件 → 复现 → 修复）
| 维度  | 说明                                                                                         |
| --- | ------------------------------------------------------------------------------------------ |
| 原理  | Source（`location.hash`）数据未净化直接写入Sink（`innerHTML`），浏览器把数据当HTML解析，攻击载荷中的事件处理器（`onerror`）被执行。 |
| 条件  | ① 页面从可控Source读数据；② 数据流入HTML解析型Sink；③ 中间无转义/净化。三者齐备即漏洞。                                     |
| 复现  | 访问 `page.html#<img src=x onerror=alert(1)>`，弹出弹窗即复现成功。                                     |
| 修复  | 用 `textContent` 替代 `innerHTML`（纯文本场景）；或用 DOMPurify 净化后再写入（需要HTML的场景）。详见标题五：防御三件套。          |
> [!important]  为什么 location.hash 特别受攻击者偏爱？
>  因为 # 后面的内容不会发送给服务端。服务端日志里看不到攻击载荷，WAF（Web 应用防火墙）也拦截不到——整个攻击只在浏览器里发生。这就是 DOM XSS 比传统 XSS 更隐蔽的原因。

# 4.更多真实载荷形态
下面这些都能突破 innerHTML 这类 Sink：

| 载荷 | 触发原理 |
| ---- | ---- |
| `<img src=x onerror=alert(1)>` | 图片加载失败触发 `onerror`（最经典） |
| `<svg onload=alert(1)>` | SVG 元素加载即触发 `onload` |
| `<script>alert(1)</script>` | ⚠️ 注意：通过 `innerHTML` 插入的 script 不会执行！这是 HTML 规范的安全设计。所以攻击者改用 img/svg 等带事件处理器的方式。 |
| `<a href="javascript:alert(1)">点我</a>` | `javascript:` 伪协议，用户点击时执行 |
| `<iframe src="javascript:alert(1)"></iframe>` | iframe 伪协议执行（部分浏览器） |
重要细节：很多人以为「\<script> 标签会执行」——错！通过 innerHTML 插入的 \<script> 不会运行（规范如此，防止 innerHTML 成为超级 eval）。但这不意味着 innerHTML 安全——\<img onerror>、\<svg onload> 照样能执行任意 JS。所以「我用 innerHTML 但过滤了 script 标签」是无效防御。
# 5.防御三件套
修复 DOM XSS 有三个层次，从基础到进阶：
## 5.1 第一件：textContent 替代 innerHTML
当只是想显示纯文本（不需要 HTML 标签生效），永远用 textContent 而非 innerHTML。textContent 会把内容当纯文本，所有 <> 都被原样显示，不解析成 HTML。
## 5.2 第二件：DOMPurify 净化
当确实需要显示富文本（比如评论支持加粗、链接），不能直接用 textContent（会显示原始标签）。这时用 [DOMPurify](https://github.com/cure53/DOMPurify)——业界标准的 HTML 净化库，它保留安全标签、剥离所有事件处理器和恶意属性。
```js
// ⭐ DOMPurify 用法（需引入 DOMPurify 库）
// <script src="dompurify.min.js"></script>
const dirty = location.hash.substring(1);
const clean = DOMPurify.sanitize(dirty);   // ⭐ 净化：保留 <b><a> 等，去掉 onerror/javascript:
document.getElementById('greeting').innerHTML = clean;   // 现在安全了
```
**⭐别自己造轮子写净化器**：HTML 解析有很多边角案例（\<a href="java script:...">、Unicode 编码、嵌套实体……）。自写的正则过滤几乎必然有绕过。永远用 DOMPurify 这类经过社区审计的库。这和上节课「别用正则做 AST 的活」是同一个道理。
## 5.3 第三件：CSP（内容安全策略）纵深防御
CSP（Content Security Policy，内容安全策略）是 HTTP 响应头，给浏览器一份「白名单」，限制能执行的脚本来源。即使代码漏了，CSP 也能挡住内联脚本和外域脚本。
```js
//  一个强 CSP 示例（HTTP 头或 meta 标签）
Content-Security-Policy: script-src 'self' https://cdn.example.com; object-src 'none'

// 含义：
// script-src 'self' —— 只允许同源脚本
// 没写 'unsafe-inline' —— 禁止所有内联脚本（含 onerror 等事件处理器）
// object-src 'none' —— 禁止 Flash/插件
```
**⭐CSP 是「纵深防御」不是「首选防御」**：CSP 能挡住 onerror=alert(1) 这类内联事件（因为禁了 inline），但如果漏洞点是 eval() 这种本就允许的执行方式，CSP 默认拦不住。所以先修代码（用 textContent/DOMPurify），再用 CSP 兜底，别指望 CSP 一个扛所有。
# 6.核心要点

| 概念                   | 一句话                                                      |
| -------------------- | -------------------------------------------------------- |
| DOM XSS 本质           | Source（可控数据）→ Sink（危险出口）无净化 = 漏洞                         |
| 与传统 XSS 区别           | 服务端响应不含恶意内容，漏洞纯在客户端 JS                                   |
| 常见 Source            | location.hash/search、referrer、postMessage、localStorage   |
| 常见 Sink              | innerHTML、document.write、eval、setTimeout(string)、$(html) |
| ⭐ innerHTML 的 script | 插入的 `<script>` 不执行，但 img/svg 事件处理器会执行                    |
| 防御① textContent      | 纯文本场景，替代 innerHTML                                       |
| 防御② DOMPurify        | 富文本场景，净化后再 innerHTML                                     |
| 防御③ CSP              | 纵深防御，禁 inline 脚本兜底                                       |
