

> 来源：用户提供的直播截图整理。以下内容仅用于合法授权范围内的 SRC、企业内测或自有环境安全测试。

## 核心原则

- 所有测试优先放在 SRC 平台、授权项目、测试账号和可控业务数据中完成。
- 高价值漏洞通常来自业务交互，而不是单个接口本身：大号/小号、买家/卖家、用户/商家、客服/普通用户之间的权限边界最容易出问题。
- 抓包时同时看请求参数和返回值，重点关注 `uid`、`userId`、`activityId`、`couponId`、`redPacketId`、`orderId`、`addressId`、`type`、`status`、`code`、`date` 等字段。
- 不要只看页面上能点到的功能，活动页、隐藏页、历史记录、第三方绑定、补签、保证金、红包、优惠券等边缘业务更容易出现逻辑漏洞。
- 返回值可控不一定就是漏洞，必须验证是否真的造成业务结果变化，例如手机号是否真的修改成功、优惠券是否真的到账、订单是否真的被越权操作。

## 1. 签到、补签、活动类接口

### 关注点

- 签到接口是否支持重复提交。
- 签到类型 `type` 是否能被修改，例如把 `type: 1` 改成其他值后是否出现无限签到、补签或异常奖励。
- 补签接口是否允许补签任意历史日期。
- 活动页是否存在隐藏活动 ID，能否通过修改活动 ID 进入未公开活动。
- 活动奖励、优惠券、积分是否只靠前端判断领取资格。

### 测试思路

- 对签到接口做短时间**并发**，观察是否出现重复签到或重复发奖。
- 修改签到类型、补签日期、活动 ID，观察服务端是否做了权限、时间和资格校验。
- 从正常活动页面的请求中提取活动 ID，再尝试枚举或替换同类 ID，检查是否能进入隐藏活动或领取隐藏奖励。

### 可能影响

- 无限签到。
- 任意补签。
- 未授权领取优惠券、积分、活动奖励。
- 访问内部测试活动或未公开活动。

## 2. 第三方绑定与账号绑定

### 关注点

- 手机、微信、QQ、微博等第三方绑定流程中是否传入 `uid`。
- 绑定接口是否校验当前登录态与被绑定账号的一致性。
- 是否能把别人的账号绑定到自己的第三方登录账号上。

### 测试思路

- 使用 A、B 两个自有账号测试绑定流程。
- 抓取绑定请求，观察是否包含用户 ID、第三方 openid、unionid、手机号等字段。
- 在授权环境中替换目标用户 ID，验证是否出现 UID 越权或账号接管风险。

### 可能影响

- 第三方登录越权。
- 账号绑定到错误主体。
- 账号接管。

## 3. 加密、防重放与参数变形

### 关注点

- 参数看起来被加密、签名或混淆时，服务端是否真的校验。
- 随机字符串形式的 ID 是否可以被替换为纯数字 ID。
- 防重放是否只在前端或网关层做了弱校验。

### 测试思路

- 对同一请求进行重复提交，观察是否存在重放成功。
- 修改 ID 形态，例如把复杂（乱码） ID 替换成 `101`、`1001`、`10001` 这类简单数字，观察后端是否接受。
- 对加密参数、签名参数、空格、编码、大小写、重复参数等边界做小范围验证。

### 可能影响

- 防重放失效。
- 绕过签名或弱校验。
- 通过参数变形访问本不该访问的数据。

## 4. 商城、浏览历史与推荐数据

### 关注点

- 访问商品详情、视频详情、内容详情时是否上传 `uid`。
- 浏览历史、推荐系统、足迹记录是否根据传入 UID 写入或读取。
- 商品、视频、内容推荐是否可被他人 UID 污染。

### 测试思路

- A 账号访问商品或视频，抓取是否传入 UID。
- 将 UID 替换为 B 账号，观察是否能写入 B 的浏览记录。
- 检查 B 账号是否出现 A 操作产生的浏览历史、推荐结果或足迹数据。

### 可能影响

- 越权读取浏览历史。
- 越权写入他人历史记录。
- 大数据推荐污染。
- 隐私泄露。

## 5. 红包、优惠券与营销活动

### 关注点

- 红包、优惠券、抽奖、满减活动是否存在独立 ID。
- 活动弹窗没有出现时，是否能从数据包中找到对应优惠券或活动 ID。
- 发红包、领红包、支付状态、领取资格是否全部由服务端校验。
- 红包或优惠券 ID 很长时，不要直接放弃，先找触发接口和返回包。

### 测试思路

- A 账号进入发红包流程，观察未支付状态下是否已经生成红包 ID。
- B 账号尝试领取一个正常可领取红包，再替换为 A 账号未支付环节生成的红包 ID。
- 检查服务端是否严格判断红包支付状态、归属、领取次数、领取人资格。
- 对优惠券弹窗、活动页、领取页中的 ID 做遍历和关联分析，寻找隐藏券或隐藏活动。

### 可能影响

- 未支付红包被领取。
- 越权领取优惠券。
- 领取隐藏活动奖励。
- 重复领取营销资产。

## 6. 修改手机号与验证码校验

### 关注点

- 修改手机号、换绑手机号、验证码校验流程是否依赖返回值。
- 返回包中的 `code`、`status`、`msg` 被修改后，前端是否直接放行。
- 前端放行后，服务端是否真的完成手机号修改。

### 测试思路

- 用自有账号走完整换绑流程，抓取验证码校验接口和下一步接口。
- 修改返回值为成功状态，例如 `code = 0` 或 `status = success`，观察前端是否进入下一步。
- 最终必须验证手机号是否真的换绑成功。只有真实换绑成功或影响核心流程，才算有效漏洞。

### 可能影响

- 验证码校验绕过。
- 手机号越权换绑。
- 账号接管。

## 7. 地址、订单与交易状态

### 关注点

- 地址 ID、订单 ID、收货 ID 是否在请求或返回值中出现。
- 地址 ID 被加密时，是否能通过修改返回值里的 ID 影响后续操作。
- 买家确认收货、卖家代发货、订单管理等流程是否校验订单归属。

### 测试思路

- 使用 A、B 两个自有账号分别创建地址和订单。
- 抓取修改地址、删除地址、默认地址、订单详情、确认收货、发货等接口。
- 替换地址 ID、订单 ID 或返回值中的 ID，验证是否能操作非本人资源。
- 在买家/卖家两个身份间交叉测试：卖家代发货是否越权，买家确认收货是否能影响他人订单。

### 特殊思路：返回值 ID 替换

有些 APP 的地址 ID、订单 ID 在请求里看起来是加密或不可猜的，但前端加载详情页时会先从接口返回包里取出一个资源 ID，再把这个 ID 带到“修改地址”“删除地址”“设为默认”等后续接口。

测试时可以重点看这个链路：

1. A 账号进入地址列表或地址详情页，抓取加载地址的返回包。
2. 观察返回包里是否有 `id`、`addressId`、`targetId`、`encryptId`、`resourceId` 等字段。
3. 在授权测试账号之间，把返回包里的资源 ID 替换为 B 账号的地址 ID。
4. 回到 APP 点击“修改地址”等后续操作，验证服务端是否仍按当前登录账号校验资源归属。

这个点的本质不是“加密 ID 被破解”，而是**前端信任了被篡改的返回值，后端在后续修改接口没有重新校验资源归属**。报告里可以写成“返回值对象引用污染导致地址越权修改”或“后续接口缺少资源归属校验导致 IDOR”。

### 可能影响

- 修改他人地址。
- 查看他人订单。
- 越权确认收货。
- 越权发货或代发货。

## 8. 保证金、余额与资金相关功能

### 关注点

- 商家保证金、余额、充值、提现、退款等资金状态是否只依赖前端显示。
- 充值渠道、回收保证金渠道是否存在状态绕过。
- 金额、订单号、支付状态、商家 ID 是否可控。

### 测试思路

- 只在授权测试环境或平台允许的范围内验证，不碰真实资金风险操作。
- 观察保证金充值、解冻、退回、提现流程中的状态字段。
- 检查是否能通过其他渠道触发保证金充值、退回或状态变更。

### 可能影响

- 保证金状态绕过。
- 未支付变已支付。
- 资金类越权。
- 商家权益异常开通。

## 9. 聊天、客服与用户互动

### 关注点

- 只开放客服聊天的场景，是否能通过改 UID 或构造会话 ID 发起普通用户聊天。
- 客服、商家、买家、普通用户之间是否存在身份边界。
- 返回值中的会话 ID、目标 UID、权限字段是否被前端信任。

### 测试思路

- 找到客服聊天的正常请求。
- 观察是否传入目标 UID、会话 ID、角色类型。
- 用自有账号替换 UID 或构造会话 ID，验证是否能越权发起聊天、查看会话或发送消息。

### 可能影响

- 越权发起会话。
- 查看他人聊天。
- 冒用客服或商家身份互动。

## 10. 大号、小号与 AB 交互测试法

### 适用场景

- 用户 A 与用户 B 的关注、私信、红包、优惠券、订单、地址、绑定、浏览记录。
- 买家与卖家的发货、收货、退款、订单管理。
- 商家与用户的活动发放、红包发放、优惠券发放。

### 方法

- 多去测试有交互的地方，最容易出现 SRC 认可的业务漏洞。
- 因为 A、B 都是自己的账号，测试过程中的所有操作都可以留证和复现。
- 重点看“谁发起”“谁接收”“资源属于谁”“服务端按谁的身份校验”。

## 常见参数观察清单

| 类型 | 常见字段 | 重点问题 |
| --- | --- | --- |
| 用户身份 | `uid`、`userId`、`memberId`、`sellerId`、`buyerId` | 是否可替换为他人 ID |
| 活动营销 | `activityId`、`couponId`、`redPacketId`、`prizeId` | 是否能领隐藏活动或重复领取 |
| 订单交易 | `orderId`、`addressId`、`payStatus`、`status` | 是否校验归属和状态 |
| 验证流程 | `code`、`status`、`token`、`verifyId` | 是否能绕过验证码或步骤校验 |
| 行为数据 | `historyId`、`itemId`、`videoId`、`goodsId` | 是否能读写他人行为记录 |
| 类型开关 | `type`、`scene`、`role`、`source` | 是否能切换到高权限或隐藏场景 |

## 漏洞评级参考

- 高危：账号接管、手机号换绑成功、第三方绑定越权、资金状态绕过、订单越权操作、未支付红包或优惠券被领取。
- 中高危：越权查看订单、地址、聊天记录、浏览历史等敏感数据。
- 中危：越权写入浏览历史、污染推荐、重复签到、重复领取低价值奖励。
- 低危或无效：只修改了前端显示或返回值，但服务端最终业务状态没有变化。

## 复盘总结

直播里的核心思路可以概括为：围绕"身份、资源、状态、次数、时间、隐藏入口"六个维度做业务逻辑测试。真正有价值的点不是单纯改包，而是证明服务端没有正确校验当前操作者是否有权访问或改变某个业务资源。
